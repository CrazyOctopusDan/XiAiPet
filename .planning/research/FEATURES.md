# Feature Research

**Domain:** 微信宠物烘焙双端小程序 + 单店即时/预约履约
**Researched:** 2026-04-16
**Confidence:** HIGH

## Table Stakes

### Customer Commerce

- 首页展示店铺品牌、Banner、主入口和会员权益，用户能快速进入下单链路。
- 商品列表需要支持分类浏览、搜索、库存状态展示、规格/配方选择和价格展示。
- 购物车需要支持跨页面同步、数量调整、库存上限校验和清空能力。
- 商品详情需要补足图文、规格、等级门槛与分享能力，否则高客单定制商品转化会受阻。

### Checkout and Fulfillment

- 订单确认必须支持履约方式切换、地址/电话维护、预约时间选择和费用明细。
- 微信支付是首发闭环的默认支付能力；余额支付是提高复购和客服处理效率的重要补充。
- 订单列表与订单详情是支付后的最小闭环，没有它们用户无法确认履约状态。

### User Account and Retention

- 个人中心需要集中展示头像、余额、会员等级和累计消费，承接复购心智。
- 地址管理、宠物资料、备注历史和余额流水是宠物定制场景的高频基础能力，不是“nice to have”。

### Merchant Operations

- 商户端至少要覆盖订单处理、商品管理、品类管理和用户余额调整，否则业务运营无法闭环。
- 店铺位置、配送费规则、会员阈值、定制提醒等运营参数需要可配置，而不是写死在代码里。

## Differentiators

- 宠物信息绑定到订单，帮助店主按宠物属性与过敏源做定制沟通。
- 商品“规格配方信息”与快速购买卡片提升列表页加购效率，适合多规格宠物烘焙商品。
- 会员等级门槛直接限制可购商品，形成明确的复购和会员成长机制。
- 预约时间与配送费联动，能把“是否能按时送达”提前在下单前说清楚，减少人工沟通。
- 余额体系结合余额流水与商户端余额调整，为售后、充值和退款提供统一账本。

## Anti-Features

- 首发就做优惠券、满减、拼团、裂变分享任务 — 会把交易闭环拖入营销系统复杂度。
- 多店、多仓、多骑手调度 — 当前是单店宠物烘焙业务，不应超前设计成平台系统。
- 过早引入跨端框架或三端统一 UI 库 — 当前最重要的是微信生态内稳定交付，而不是表面复用率。
- 把订单状态、配送费、余额扣减放到前端逻辑里 — 这不是提效，是直接制造财务和履约风险。

## Complexity Notes

| Feature Area | Complexity | Why |
|--------------|------------|-----|
| 商品浏览与详情 | Medium | 页面多但模式稳定，主要是分类联动、规格选择和库存态 |
| 购物车 | Medium | 需要跨页面状态同步和规格商品建模 |
| 结算与预约 | High | 同时涉及履约方式、地址、电话、预约规则、宠物、备注、配送费 |
| 支付与订单 | High | 涉及支付回调、订单快照、库存与余额一致性 |
| 商户运营 | Medium | CRUD 为主，但字段多、表单复杂、需权限隔离 |
| 余额体系 | High | 涉及可审计账本、退款/充值/消费流转 |

## Dependencies Between Features

- 商品浏览依赖商品、品类、配置数据先建模。
- 购物车依赖商品规格建模与库存字段统一。
- 结算依赖地址、宠物、用户手机号、店铺配置和配送费规则。
- 支付依赖订单快照、金额计算和云函数敏感写入路径。
- 商户端商品管理依赖前台商品展示字段已经统一，否则双端模型会反复返工。
- 余额调整依赖流水账本设计，否则无法追责和对账。

## Recommendations for v1 Scope

### Must Include in v1

- 客户端：首页、商品列表、搜索、详情、购物车、确认订单、支付、订单、个人中心、地址、宠物、余额流水
- 商户端：订单管理、品类管理、商品管理、用户查询与余额调整、基础运营配置
- 后端：商品、订单、用户、地址、宠物、配置、余额流水、库存与支付敏感流程

### Defer to v2+

- 优惠券和促销引擎
- 用户评价/晒单
- 物流轨迹和消息通知中心
- 多门店/多商家模式
- Web 商家后台与经营分析大屏

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| merchant miniapp -> Cloud Functions | Untrusted operator input enters privileged order query/update handlers |
| Cloud Functions -> orders collection | Fulfillment mutations cross into backend-owned order records and audit state |
| shared status helpers -> customer pages | Incorrect label mapping can misrepresent order progress across client and merchant surfaces |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01-01 | E | `queryMerchantOrders` / `getMerchantOrderDetail` / `updateMerchantOrderStatus` | mitigate | Require merchant access on every handler via the existing merchant auth context, never direct DB reads from the miniapp, and project only merchant-safe fields. |
| T-06-01-02 | T/R | `updateMerchantOrderStatus` | mitigate | Enforce D-06 transition graph, lock terminal states, and append operator/timestamp/before-after/reason/adjustment-method audit entries for every mutation. |
| T-06-01-03 | I | customer + merchant status labeling | mitigate | Centralize fulfillment labels in shared helpers and cover merchant/customer parity in automated tests so user-visible state cannot diverge silently. |
</threat_model>

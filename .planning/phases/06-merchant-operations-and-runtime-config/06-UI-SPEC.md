## Phase Visual Direction

- 商户端不是传统固定 tab 后台，也不是桌面 SaaS 风格。主气质必须是“微信内轻量工作台”: 大圆角、奶油底色、明确卡片入口、信息密度中等、操作路径短。
- 保留客户端的暖黄、奶油、棕色文字和宠物品牌感，但减少插图泛滥。插图只允许出现在管理首页四张入口卡片里，订单、商品、用户、配置页面转为更克制的线性图标与信息卡片。
- 禁止使用冷蓝灰为主色、密集网格表格、永久左侧导航、通栏数据大表。列表以卡片和分组块表达，不做桌面后台降级版。
- 页面背景统一使用暖色渐变或纯奶油面；核心内容始终落在白奶油卡片上，避免整页纯白造成品牌断裂。

## Order Status Contract

| Scenario | Display Rule | Primary Action | Audit Requirement |
|----------|--------------|----------------|-------------------|
| 配送订单 | 依次展示 `待处理` → `制作中` → `配送中` → `已完成` | `更新订单状态` | 每次修改都记录操作人、时间、前后状态、备注 |
| 自取订单 | 依次展示 `待处理` → `制作中` → `待自取` → `已完成` | `更新订单状态` | 同上 |
| 快递订单 | 依次展示 `待处理` → `制作中` → `待发货` → `已完成` | `更新订单状态` | 同上 |
| 未支付人工兜底 | 卡片和详情页同时显示 `待支付` 次级标识，不改变“按履约进度分组”的主结构 | `标记已支付/已处理` | 必填 `调整方式`、`原因备注`，并保留原支付状态痕迹 |
| 终态订单 | `已完成`、`已取消` 使用只读状态块，不提供回退入口 | none | 仅展示历史记录，不可改写 |

Presentation rules:
- 客户端与商户端使用同一套细分状态文案，不允许再做“用户态简化字样”。
- 同一页面内只能存在一个主状态胶囊； `待支付`、`异常` 之类只作为次级徽标。
- 状态切换入口只出现在详情页，不在列表卡片上直接提供多按钮操作。

## Product Editing Contract

| Step | Fields | Layout Rule | Validation Feedback |
|------|--------|-------------|---------------------|
| 基础信息 | 商品图、名称、描述、一级品类、会员等级、库存 | 图片先于文本字段；库存和等级放在同一卡片内；名称和描述不拆多列 | 字段错误在控件下方显示 14px 红字，不只 toast |
| 规格配方与价格 | 基准价格、规格行、配方标题、配方行、组合价覆盖、限购数量、详情内容 | 规格和配方都用可增删的行编辑器；每一行必须有标题、加价和删除入口；组合覆盖价放在独立“少量例外”卡片 | 自动计算价实时展示；存在手动覆盖时显示 `已覆盖自动计算` 标记 |
| 上架设置 | 状态、可用履约方式、售罄/库存提示、预览摘要 | 该步骤先展示面向商户的发布检查清单，再放开关和模式多选 | 未选择任何履约方式时不能保存商品 |

- 商品首版只挂一个一级品类，字段组件必须是单选，不得误导为多选。

- 新建商品第一步保存成功后才生成后续步骤上下文，不在空白二三步中先展示无效组件。

## User Detail And Balance Contract

- 用户详情页的视觉焦点是“当前余额 + 最近操作”，不是资料编辑页。
- 余额卡片必须把当前余额做成 28px/600 显示，下方用 14px 元信息展示会员等级和最近更新时间。
- `调整余额` 抽屉使用三段动作切换：`增加余额`、`扣减余额`、`改为指定余额`。三者权重相同，不折叠到二级菜单。
- 选择 `扣减余额` 或 `改为指定余额` 时，抽屉底部必须即时显示“调整后余额”预览；如果结果小于 0，直接禁用主按钮并显示错误说明。
- 原因类型用单选胶囊，固定为 `充值`、`补偿`、`人工纠错`、`线下收款`、`其他`。
- 二次确认文案固定为 `请确认本次余额调整，提交后将生成流水记录。`
- 用户侧账本文案对商户动作统一标准化为 `余额调整` 主标题 + 简短备注，不回显后台长备注全文。

## State And Feedback Contract

| State | Contract |
|-------|----------|
| Loading | 首页卡片、订单卡片、用户卡片、配置分区都使用骨架屏；骨架块保持最终布局形状，不用通用灰条占位。 |
| Empty | 每个模块都使用一个主标题 + 一行说明 + 一个回流动作，不超过三行文案。订单空态回流到管理首页；商品空态主动作是 `新建商品`；用户空态不给按钮，只提示修改关键词。 |
| Save success | 先局部按钮 loading，再在卡尾显示 `已保存 HH:mm`，同时 toast `保存成功`；禁止只 toast 不改页面状态。 |
| Validation error | 优先字段级提示，其次卡片顶部错误摘要；表单错误不只用 toast。 |
| Destructive warning | 删除或高风险余额调整时，弹层正文必须明确后果，不使用 `是否继续？` 这种模糊文案。 |
| Audit visibility | 任何涉及订单状态和余额调整的结果页，都必须能直接看到最近一次操作人、时间和备注。 |

# Phase 11: OSS Asset Migration and Upload Flow - Research

**Researched:** 2026-05-11
**Status:** Complete

## Research Question

What needs to be true to plan Phase 11 well: replacing CloudBase storage with Alibaba Cloud OSS for product and runtime-config assets, with controlled merchant uploads, miniapp-compatible crop/compression, role-specific image variants, idempotent legacy reference reporting, and cost-aware public display URLs?

## Executive Summary

Use backend-issued OSS PostObject form policies for merchant direct upload. The miniapp should never receive long-lived OSS credentials. `apps/api` should generate scoped object keys, short-lived policy fields, and a signed confirm token. The merchant miniapp uploads local files to OSS with `wx.uploadFile`, then confirms with the API so MySQL stores an `OssAssetReference` shape.

For compression, prefer miniapp-side role-driven processing before upload: `wx.chooseMedia` for selecting compressed image candidates, `wx.cropImage` when the base library supports the required ratio, then `wx.compressImage` with role-specific width/height and quality. OSS image styles can remain a fallback/supplement, but list/card views should persist and read lightweight URLs rather than dynamically resizing large originals per request.

Store role-specific final URLs and metadata, not original source images by default. Use `product-cover`, `product-introduction`, `product-detail`, and `runtime-banner` as first-class roles.

## Official API Findings

### WeChat Mini Program Upload and Image Processing

- `wx.uploadFile` sends a HTTPS `multipart/form-data` POST with `url`, local `filePath`, `name`, optional `header`, optional `formData`, and returns `statusCode` plus string `data`. This matches OSS PostObject form upload because the miniapp can include policy/signature fields in `formData` and the image file as the `file` field.
- `wx.chooseMedia` is supported from base library `2.10.0`, supports image-only selection with `mediaType: ['image']`, and can request `sizeType: ['compressed']`. It returns `tempFiles` with `tempFilePath`, `size`, and image dimensions when available.
- `wx.cropImage` is supported from base library `2.26.0`, is callback-only, and supports fixed ratios: `16:9`, `9:16`, `4:3`, `3:4`, `5:4`, `4:5`, and `1:1`. This is enough for role-driven crop UI, but the plan needs a fallback for older bases.
- `wx.compressImage` is supported from base library `2.4.0`, supports `quality` from 0 to 100, and supports `compressedWidth`/`compressedHeight` from `2.26.0`. Official docs note that iOS compression only supports JPG, so the implementation should validate post-compression file size and not rely on PNG compression behaving the same.

### Alibaba Cloud OSS Upload and Image Delivery

- OSS PostObject/browser form upload uses a policy and signature generated by the application server. This fits the phase security boundary: `apps/api` holds OSS access keys, returns only short-lived upload form fields, and scopes object keys to merchant-controlled asset prefixes.
- CORS must allow the miniapp upload origin/requests to call the OSS bucket endpoint. The release docs should state the required bucket CORS methods, headers, and WeChat legal upload/download domain setup.
- Public product and banner images should be served by OSS or a future CDN/custom asset domain, not proxied through `apps/api`.
- OSS image processing styles are useful for deterministic resize/crop transforms and future CDN integration. For this phase, persisted role-specific final URLs are preferred for list/card surfaces to avoid accidentally loading large originals and paying for unnecessary traffic.

## Recommended Asset Contract

Add shared asset types in `packages/shared/src/types/assets.ts`:

```ts
export type OssAssetRole =
  | 'product-cover'
  | 'product-introduction'
  | 'product-detail'
  | 'runtime-banner';

export type OssAssetVariantName = 'thumbnail' | 'display' | 'detail' | 'banner';

export interface OssAssetVariant {
  name: OssAssetVariantName;
  objectKey: string;
  url: string;
  width: number;
  height: number;
  sizeBytes: number;
  contentType: 'image/jpeg' | 'image/png' | 'image/webp';
}

export interface OssAssetReference {
  provider: 'oss';
  role: OssAssetRole;
  bucket: string;
  region: string;
  objectKey: string;
  url: string;
  variants: OssAssetVariant[];
  width: number;
  height: number;
  sizeBytes: number;
  contentType: 'image/jpeg' | 'image/png' | 'image/webp';
  uploadedAt: string;
}
```

The existing string fields can be kept during migration for compatibility:

- `imageFileId`: canonical object identity, e.g. `oss://{bucket}/{objectKey}`
- `imagePreviewUrl`: lightweight display URL for legacy callers
- New Product JSON fields: `imageAsset`, `introductionImageAssets`, `detailImageAssets`
- Runtime config banner value: `{ asset: OssAssetReference, fileId: string, altText: string }` during compatibility transition

## Role Rules

| Role | Crop Ratio | Final Variants | Target Limits | Display Surface |
|------|------------|----------------|---------------|-----------------|
| `product-cover` | `1:1` | `thumbnail` 480x480, `display` 960x960 | thumbnail <= 180 KB, display <= 450 KB | product list/card/detail first image |
| `product-introduction` | `4:3` | `display` 960x720 | <= 500 KB | product introduction section |
| `product-detail` | `3:4` | `detail` 960x1280 | <= 700 KB | product detail long image section |
| `runtime-banner` | `16:9` | `banner` 1280x720 | <= 650 KB | home banner |

The executor may tune exact values if tests show current UI sizes need adjustment, but the plan should keep these values concrete so upload validation and tests are verifiable.

## Backend Design

Add `apps/api/src/modules/assets`:

- `policy.ts`: object key generation, policy/signature creation, signed confirm token creation/verification.
- `service.ts`: role validation, merchant authorization boundary, upload authorization response, confirm validation, public URL construction.
- `repository.ts`: update product/runtime config asset JSON through existing catalog/runtime-config repositories where possible.

Add `apps/api/src/routes/merchant/assets.ts`:

- `POST /api/v1/merchant/assets/upload-policy`
- `POST /api/v1/merchant/assets/confirm`

The route depends on `requireMerchantSession`. The service returns:

```ts
{
  ok: true,
  upload: {
    method: 'POST',
    url: 'https://{bucket}.{endpoint}',
    fileFieldName: 'file',
    objectKey: 'merchant/{merchantId}/assets/{role}/{yyyy}/{assetId}-{variant}.jpg',
    expiresAt: '2026-05-11T10:00:00.000Z',
    formData: {
      key: '...',
      policy: '...',
      OSSAccessKeyId: '...',
      Signature: '...',
      success_action_status: '200',
      'x-oss-object-acl': 'public-read'
    },
    confirmToken: '...'
  }
}
```

The confirm request should include `confirmToken`, `role`, `objectKey`, `url`, dimensions, size, content type, and variant name. The backend validates token/key/role/expiry and returns the persisted asset reference.

## Miniapp Design

Add `apps/merchant-miniapp/src/services/assets.ts`:

- `selectAssetSource(role, mode)`: `mode` is `processed` or `miniapp`.
- `processImageForRole(role, filePath)`: calls `wx.cropImage` where available, then `wx.compressImage`; validates file size/dimensions.
- `requestUploadPolicy(role, variantName, metadata)`: calls merchant API.
- `uploadFileToOss(policy, filePath)`: wraps `wx.uploadFile`.
- `confirmAssetUpload(policy, metadata)`: calls merchant API confirm.
- `uploadMerchantAsset(role, options)`: returns `OssAssetReference`.

Re-enable `uploadProductImage` by delegating to `uploadMerchantAsset('product-cover')`. Runtime config banner upload should use `runtime-banner`. Product introduction/detail upload can reuse the same asset service and persist arrays without a full media library.

## Migration Strategy

CloudBase references are mostly fake/non-production. Implement a report-first script:

- Scan product `imageFileId`, product `imagePreviewUrl`, product asset JSON fields, runtime config values, and customer seed data for `cloud://`.
- Classify each as `migratable`, `placeholder`, `missingSource`, `alreadyOss`, or `skipped`.
- Write a deterministic JSON report to `apps/api/tmp/asset-migration-report.json` or a path supplied by CLI argument.
- Only copy files when an explicit local source map/export is supplied; otherwise report and skip.

## Validation Architecture

Use existing Vitest suites and TypeScript checks. Because this phase modifies Prisma schema, shared schemas, API routes, and both miniapps, validation must include:

- API: `pnpm --filter @xiaipet/api test`, `pnpm --filter @xiaipet/api typecheck`, `pnpm --filter @xiaipet/api build`
- Shared: `pnpm --filter @xiaipet/shared test`, `pnpm --filter @xiaipet/shared typecheck`
- Merchant miniapp: `pnpm --filter @xiaipet/merchant-miniapp test`, `pnpm --filter @xiaipet/merchant-miniapp typecheck`, `pnpm --filter @xiaipet/merchant-miniapp build`
- Customer miniapp: `pnpm --filter @xiaipet/customer-miniapp test`, `pnpm --filter @xiaipet/customer-miniapp typecheck`, `pnpm --filter @xiaipet/customer-miniapp build`
- Audit: `rg "ASSET_UPLOAD_PENDING_OSS|wx\\.cloud\\.getTempFileURL|uploadCloudFile" apps/customer-miniapp apps/merchant-miniapp apps/api packages/shared` should return no production matches after Phase 11.
- Schema: `pnpm --filter @xiaipet/api db:generate` after Prisma schema updates; a live DB migration/push is a blocking execution task when a local MySQL/RDS target is available.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| OSS credentials leak to miniapp | High | Return only short-lived policy fields; never include `OSS_ACCESS_KEY_SECRET`; add tests and logger redaction. |
| Merchant confirms an arbitrary object key | High | Backend generates keys and validates `confirmToken`, role, merchant prefix, object key, expiry, and metadata limits. |
| List/card screens load full detail images | Medium | Persist `thumbnailUrl`/variant URLs and update customer display code to use the smallest role-appropriate URL. |
| `wx.cropImage` unavailable on old base library | Medium | Fallback to processed-upload validation and `wx.compressImage` where possible; show actionable toast. |
| iOS PNG compression not effective | Medium | Prefer JPG output naming/content type for processed assets; validate post-compression size and reject oversized files. |
| Old fake CloudBase references distract execution | Low | Report-first migration, skip missing fake files, replace seed defaults with OSS/local placeholders. |

# 订单状态事件历史设计

## 目标

商户端订单详情必须展示真实、按时间排序的订单事件。每一次履约状态变化均保留独立时间、操作人和备注；后续操作不得覆盖既有事件。

## 已确认范围

- 新订单及本次发布后的状态变更记录完整事件历史。
- 事件涵盖订单创建、支付完成、商户状态变更、客户确认完成、客户取消、商户取消和系统自动完成。
- 旧订单不伪造缺失的中间状态。详情继续展示可证实的创建、支付、取消和当前状态事件。
- 列表卡片右上角仍固定显示下单时间，不受状态事件影响。

## 方案比较

1. 在 `Order.merchantOverride` 内维护事件数组：不需要迁移，但每次写入会改写整份 JSON，难以索引、并发安全性差，也无法防止后续代码覆盖历史。
2. 从订单的 `updatedAt` 等字段推断：无迁移，但只能得到最后状态，不能满足多次流转审计。
3. 新建关联的 `OrderStatusEvent` 表：每次状态变化追加一行，可按订单和时间查询，并可与状态更新同事务提交。采用此方案。

## 数据模型

`OrderStatusEvent` 关联一个订单，保存：事件类别、变更前/后订单状态、支付状态和履约状态、操作人类型/标识/名称、备注、发生时间。事件以创建时间和自增稳定顺序倒序读取。

状态更新与事件写入必须在同一个 Prisma 事务中进行：订单更新成功但事件写入失败时，整次操作回滚，反之亦然。

## 读写流程

订单详情 API 优先读取事件表。若订单没有事件记录，则保留现有基于订单字段的只读兜底，以确保旧订单仍可展示已知关键时间。

所有改动订单状态的入口必须经由统一仓储方法写入事件，包含支付结算、商户更新、客户确认完成、客户取消和自动完成。商户提交的操作人和备注写入事件，并在时间线展示；支付和自动处理写为系统事件。

## 验收与测试

- 多次履约状态更新后，详情 API 返回每次变化，且包含正确的前后状态与时间。
- 支付、取消、客户确认和自动完成各自产生一次事件。
- 旧订单无事件记录时仍展示可证实的关键事件，且不生成虚假的中间状态。
- 商户端按 API 时间线展示完整记录，并继续固定使用 `createdAt` 作为订单卡片时间。
